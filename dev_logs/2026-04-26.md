  - `dicom2` 的 mask 体素规模较大（典型为 `212×300×300`），在全体素空间上跑 `MCP_Geometric.find_costs()`（尤其用于 `fast_marching` 与 TEASAR tree 提取）耗时过长，导致流程没跑完就被中断。
- **修复（工程化）**：
  - 对 `astar_cost / fast_marching / teasar` 增加 **mask 包围盒 bbox 裁剪**：只在 `mask` 的 bounding box 内运行图搜索/TEASAR，从而把计算量显著降下来，保证对比流程可在合理时间内完成并产出完整 run 目录。
  - 修复后 dicom2 多路径对比成功产物示例：`output/20260426_225431/`（含 `run_metrics.json` 与 HTML）。

### 经典 TEASAR 原理简介（Tree-structure Extraction by Adaptive Subdivision of Axial Rays）

- **目标**：从 3D 二值/概率体数据中提取“树状管腔结构”的中心线（例如血管/气道树），并自然得到分支拓扑。
- **核心思想（直观版）**：
  - 先做距离变换（每个体素到边界的距离），把它当作“局部半径/通道粗细”的 proxy；
  - 以“最大内切球/最粗通道”作为优先级，先抽取主干，再逐步扩展到分支；
  - 通过“轴向射线/自适应细分”与“避开已解释区域”的策略，避免重复覆盖同一通道，从而得到一棵中心线树。
- **为什么需要改进（结合本工程现象）**：
  - TEASAR 的有效实现通常需要更严格的“已覆盖区域管理/半径驱动的分支生长/路径回溯与平滑”，而不是简单 greedy 逐步走；
  - 否则容易出现：路径绕圈、过长、急拐（Rmin 极小）等现象，虽然某些贴壁统计可能“看上去很好”，但并未完成可用的导航主路径任务。
