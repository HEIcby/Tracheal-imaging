## 阶段性工作总结（4/13–4/26，高信号版）

### 1) 主流程与可用入口稳定化

- **形成“Windows 快速开始”口径**：根目录 `README.md` 明确推荐以 `dicom_trachea_complete.py` 作为端到端入口，从 DICOM 一键产出交互式 3D HTML。
- **关键约束显式化**：`--z-min/--z-max` 与种子（`--start-z/--start-idx`）变为必填；把“Z 裁剪与种子”作为能否跑通 3D 充气法的第一优先级旋钮。
- **输出归档工程化**：每次运行按时间戳落目录，便于对照、回溯与复跑复现。

### 2) 3D 可视化与“虚拟支气管镜”观感工程化（Plotly / PyVista 双线）

- **Plotly 内镜观感的关键工程经验**（4/24–4/26 逐步固化）：
  - 离线注入 `cam.fwd/lookDist`：减少弯道“看墙/看偏”，提升稳定性与一致性。
  - 冻结 `navCamBasis`：修复少数帧 scene basis 重算导致的相机跳变。
  - “腔外/腔内/融合”三态：用双层网格与材质/光照组合逼近 two-sided 观感。
  - **交互稳定性**：模式切换优先 `opacity≈0` 隐藏而非 `visible=false`，减少 scene 重算与卡顿；播放中任意手动操作统一 `stop()` 避免相机抢控。
- **PyVista 离线内镜 MP4**（4/23 起对接）：
  - 支持离屏渲染输出 MP4，并额外输出 `_playable.mp4`（H.264 baseline/yuv420p/faststart）以提升 Windows 播放器兼容性。

### 3) 导航线/插管路径：指标化、可对比、可复现

- **指标工程化（4/25 形成体系，4/26 加强闭环）**：
  - 路径长度 \(L\)、最小转弯半径估计 \(R_{min}\)（越大越平滑/更插管友好）
  - 贴壁风险：`d_min / d_p05 / d_med`（mm；其中 `d_p05` 作为更稳健核心指标）
  - 输出三处一致：终端摘要 + HTML 底部表格 + JSON 归档（`_navigation_metrics`）
- **对比实验能力工程化（`--nav-compare`）**：
  - 同一次运行内叠加输出多算法路径与分组指标，并输出 `selected_algorithm` 与选择规则说明。
  - 4/26 进一步明确对比口径：主路径可固定为 `skeleton_dijkstra`，避免“对照算法抢主路径”造成口径漂移；其余算法仅作对照并允许标记失败。
- **算法对比的阶段性结论（基于 4/26 记录）**：
  - `dt_ridge` 多例表现为弱基线：`Rmin` 偏小、更贴壁，作为“传统法不适用反例”保留。
  - `astar_cost / fast_marching` 在部分样本上指标更强（`Rmin`、`p05` 更优），具备后续深入价值。
  - `teasar(tree baseline)` 替换旧 greedy 后可复现跑通，但部分样本仍出现贴壁统计较差，作为“可复现对照”保留并待后续优化。

### 4) 帧率/卡顿：从“可播放”到“可演示”的转折点

- **前端播放主链路优化**（4/25–4/26）：
  - 播放改为 `requestAnimationFrame` + 基于时间的节流（以 `ms` 作为目标步进间隔）。
  - 低频 `resize` 保护触发，避免每帧重布局造成卡顿。
  - 点数降采样旋钮与 `ms=10–100` 区间，形成可复现实验入口。
- **4/26 观感口径的最终默认建议**：
  - **取点密度（插值倍数）= 1**
  - **步进间隔 = 10ms**
  - **默认模式 = 融合模式**
  - **视频模式默认速度 = 0.25×**（慢放 4 倍，演示更稳）
- **视频模式工程化**（4/26）：
  - 预计算 keyframes；播放时只读数组。
  - “视频叠加播放 + 严格同步（sync.json）+ 返回交互对齐到当前帧导航点”，把“演示流畅”和“停下可交互检查”合并为同一套产物能力。
  - `file://` 兼容：同步表在导出后回写注入 HTML，避免浏览器拦截 `fetch` 导致不同步。

## 当前遗留问题与风险清单（从既往日志收敛）

- **长耗时/“像异常退出”的误判风险**：在大体素 mask 上跑 `fast_marching/TEASAR` 等全体素图搜索可能极慢，导致只落 `run_args.json` 但没生成 `run_metrics.json/HTML`。已引入 bbox 裁剪策略缓解，但后续仍需关注极端样本耗时与超时口径。
- **对照算法的“可用性判定”**：对 `teasar` 等对照算法需要稳定的 sanity rule（例如 `Rmin` 下界、长度上界、贴壁阈值等），避免把“不可用轨迹”误当成优解。
- **帧率与交互的边界条件**：个别情况下“返回交互偶发白屏/相机控制异常”仍可能出现；已在 4/26 提到可用 `forceSceneRedraw()` 与 `relayout` 兜底作为后续加固方向。
